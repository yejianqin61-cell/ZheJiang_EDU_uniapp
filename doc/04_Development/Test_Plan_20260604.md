## 1. 项目概要

### 1.1 项目定义

**AI 智能组卷小程序** — 面向浙江省中小学教师的微信小程序，实现"输入条件 → AI 生成试卷 → 预览 → 支付 → 导出 DOCX/PDF"的完整业务闭环。

### 1.2 技术架构

```
┌──────────────────────────────────────────────────────┐
│              微信小程序 / H5 (uni-app)                 │
│          Vue 3 + Pinia + TypeScript                  │
│          17个页面 / 3个Store / 1个API层               │
└──────────────────────┬───────────────────────────────┘
                       │ HTTP/JSON (Bearer JWT)
┌──────────────────────▼───────────────────────────────┐
│            NestJS Backend (TypeScript)                │
│  ┌──────────┬──────────┬──────────┬──────────┐       │
│  │  Auth    │  User    │  Paper   │  Order   │       │
│  ├──────────┼──────────┼──────────┼──────────┤       │
│  │ Payment  │  Export  │   KB     │  Admin   │       │
│  └──────────┴──────────┴──────────┴──────────┘       │
│  TypeORM | BullMQ | JWT | Throttler                 │
└──┬──────────┬──────────┬──────────┬─────────────────┘
   │          │          │          │
   ▼          ▼          ▼          ▼
PostgreSQL  Redis   Python导出  阿里百炼AI
(pgvector)  (BullMQ) (Flask)   (Qwen+Embedding)
```

### 1.3 核心业务链路

| 链路 | 步骤 |
|------|------|
| **教师组卷链路** | 登录 → 配置条件(年级/科目/知识点/难度/题数) → AI生成 → 预览(脱敏) → 下单 → 微信支付 → 导出DOCX/PDF → 下载 |
| **管理员入库链路** | 上传文件(MD/DOCX/PDF/PNG) → OCR(图片) → AI切题 → AI打标 → 嵌入向量化 → 审核(通过/拒绝) → 入库 |
| **管理运维链路** | 仪表盘统计 → 题目管理(筛选/删除) → 文件管理 → 知识点中心 → 角色管理 |

## 3. 测试策略

### 3.1 测试金字塔

```
         ╱  E2E (10%)
        ╱   全链路业务流程
       ╱═══════════════════
      ╱   集成测试 (30%)
     ╱    跨模块交互/数据库/外部服务
    ╱═══════════════════════════
   ╱    单元测试 (60%)
  ╱     Service/Util/Guard/Entity
 ╱══════════════════════════════════
```

### 3.2 测试原则

1. **Service 全覆盖**: 所有 public 方法 ≥2 个测试用例（成功 + 主要失败路径）
2. **边界值优先**: 空输入、0值、最大值、负数、超长字符串
3. **业务错误路径**: 订单不存在、重复支付、未支付导出、题库不足等
4. **外部依赖隔离**: LLM/微信支付/COS/OCR 通过 mock 隔离，但保留 1 个"真实集成"冒烟测试
5. **前端分层测试**: Store 逻辑 → API 调用 → 组件渲染，逐层测试
6. **不重复造轮子**: 使用 `backend/src/test-utils.ts` 统一 mock 复用

### 3.3 Mock 策略

| 依赖 | 单元测试 | 集成测试 |
|------|---------|---------|
| 数据库 (TypeORM Repository) | 完全 Mock | SQL.js 内存数据库 |
| LLM API (阿里百炼) | Mock axios 返回 | Mock HTTP 服务器 |
| 微信支付 API | Mock WxPayClient | Mock HTTP 服务器 |
| 微信登录 API | Mock axios | Mock HTTP 服务器 |
| COS 对象存储 | Mock CosService | 本地文件系统 |
| OCR (Tesseract/PaddleOCR) | Mock OCRService | 真实 tesseract.js |
| Python 导出服务 | Mock axios | 真实 Flask 测试客户端 |
| Redis / BullMQ | Mock Queue | 跳过（开发模式回退） |

## 12. 风险与对策

| 风险 | 概率 | 影响 | 对策 |
|------|------|------|------|
| Windows + ts-jest + SQL.js OOM | 高 | 原有覆盖率无法收集 | 减少worker数(--maxWorkers=2)，或在WSL中运行 |
| tesseract.js 语言包下载阻塞CI | 中 | OCR测试超时 | OCR测试标记为skip CI，仅本地运行 |
| Python LibreOffice 不可用 | 高 | PDF测试无法运行 | Mock subprocess.run，仅测试DOCX+健康检查 |
| LLM API 调用成本高 | 高 | 真实调用成本不可控 | 单元测试全部mock，仅1个集成冒烟测试用真实API |
| 微信支付需要真实证书 | 高 | 生产路径无法测试 | Mock HTTP层，单独验证签名算法的数学正确性 |
| 前端uni-app组件测试工具链不成熟 | 中 | 页面测试困难 | 优先测Store和API层，组件测试用snapshot |
| 测试数据量不足 | 中 | 边界测试覆盖不全 | 编写factory函数生成各种边界数据 |
| Docker环境未就绪 | 中 | 部署E2E无法执行 | 在开发机上用docker compose up测试 |

## 总结

本测试计划覆盖了项目的 **11 个模块**，规划了约 **450 个测试用例**（在现有 56 个基础上新增约 394 个），预计 **7 个工作日**完成。

测试分为 5 个阶段递增推进：
1. **Phase 0** (基础设施) → 通用工具和mock工厂
2. **Phase 1** (后端单元测试) → 最高优先级：WxPayClient、知识库子服务、控制器层
3. **Phase 3** (集成测试) → 5条核心业务链路的跨模块集成验证
4. **Phase 4** (前端测试) → API层、Store层、核心页面组件
5. **Phase 5** (Python + 性能安全) → 导出服务验证 + 性能指标 + 安全扫描

测试完成后，预计后端语句覆盖率从 ~87% 提升至 **≥90%**，分支覆盖率从 ~82% 提升至 **≥85%**，前端和Python导出服务从 0% 达到 **60-80%** 覆盖。
